# Puppeteer MCP 기술 참조 가이드

## 개요
Puppeteer MCP(Model Context Protocol) 서버는 AI 에이전트가 브라우저를 제어하여 웹 페이지를 탐색, 스크린샷 캡처, 데이터 스크래핑을 수행할 수 있게 해주는 도구입니다.

## 주요 오류 및 원인: "Agent terminated due to error"
이 오류는 에이전트와 MCP 서버 간의 통신이 중단되었을 때 발생하며, 주요 원인은 다음과 같습니다.

1. **리소스 부족 (Memory/CPU):**
   - Chromium 브라우저는 메모리 소비가 큽니다. 시스템 리소스가 부족하면 프로세스가 강제 종료됩니다.
2. **브라우저 실행 실패:**
   - 브라우저 바이너리 경로가 잘못되었거나, 권한 문제로 실행되지 않는 경우입니다.
3. **타임아웃 (Timeout):**
   - 페이지 로딩이나 복잡한 스크립트 실행이 설정된 시간을 초과할 경우 연결이 끊깁니다.
4. **데이터 과부하:**
   - 너무 큰 HTML 소스나 대용량 로그를 한 번에 전송하려고 할 때 IPC(Inter-Process Communication) 채널이 붕괴됩니다.
5. **샌드박스 설정 문제:**
   - 특정 환경에서 브라우저 보안 샌드박스가 충돌을 일으킬 수 있습니다.

## 권장 모범 사례 및 환경 설정 (Windows 기준)

### 1. `mcp_config.json` 설정 최적화
안정성을 위해 다음과 같은 옵션을 포함하는 것이 좋습니다.

```json
{
  "mjs-puppeteer": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-puppeteer"],
    "env": {
      "PUPPETEER_SKIP_CHROMIUM_DOWNLOAD": "false"
    }
  }
}
```

### 2. 예방 조치
- **절대 경로 사용:** 사용자 환경 변수 이슈를 피하기 위해 가급적이면 `node`나 `npx`의 절대 경로를 사용하십시오.
- **단계별 수행:** 한 번에 전체 페이지를 긁기보다는 "검색 -> 결과 확인 -> 개별 페이지 방문" 순으로 태스크를 쪼개서 요청하십시오.
- **주기적 재시작:** 에이전트가 무거워지면 IDE나 에이전트 프로세스를 재시작하여 메모리를 확보하십시오.

### 3. 문제 해결 절차
- **수동 실행 테스트:** 터미널에서 `npx @modelcontextprotocol/server-puppeteer`를 직접 실행하여 라이브러리 설치 및 초기 구동에 문제가 없는지 확인합니다.
- **브라우저 캐시 삭제:** Puppeteer가 사용하는 임시 디렉토리나 브라우저 캐시가 꼬였을 경우 이를 삭제합니다.
- **보안 소프트웨어 확인:** Windows Defender나 방화벽이 `chrome.exe` 또는 `node.exe`의 네트워크/프로세스 제어를 차단하고 있지 않은지 확인합니다.
